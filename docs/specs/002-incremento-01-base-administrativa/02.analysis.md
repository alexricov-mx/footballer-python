# Spec: 002 - Incremento 01: Base Administrativa

## 2. Análisis Técnico

### Resumen

Este documento describe el enfoque técnico y las decisiones de diseño para implementar la "Base Administrativa" del proyecto Footballer. El objetivo es establecer un fundamento sólido, escalable y mantenible para los futuros incrementos.

### 1. Diseño de Modelos (`core/models.py`)

Se creará una aplicación Django llamada `core` para albergar los modelos de datos centrales. Para evitar la repetición de código y estandarizar nuestras entidades, se utilizará un modelo base abstracto.

-   **`BaseModel` (Modelo Abstracto):**
    -   Contendrá campos comunes de auditoría y estado que heredarán todos los demás modelos.
    -   Campos:
        -   `id`: `UUIDField` como clave primaria para evitar la enumeración de recursos y mejorar la seguridad.
        -   `is_active`: `BooleanField` para borrado lógico.
        -   `created_at`: `DateTimeField` (auto-poblado en la creación).
        -   `updated_at`: `DateTimeField` (auto-poblado en la actualización).

-   **Modelos de Entidades Principales:**
    -   **`Federacion`**:
        -   `nombre`: `CharField` con `unique=True`.
    -   **`Liga`**:
        -   `nombre`: `CharField`.
        -   `federacion`: `ForeignKey` a `Federacion`.
        -   Se aplicará una restricción a nivel de base de datos (`UniqueConstraint`) para asegurar que el par `(nombre, federacion)` sea único.
    -   **`Sede`**:
        -   `nombre`: `CharField`.
        -   `direccion`: `CharField`.
        -   `ciudad`: `CharField`.
        -   `referencia`: `TextField` (opcional).
        -   `foto`: `ImageField` para la carga de imágenes. Se configurará para guardarlas en un subdirectorio (ej. `sedes/`).
    -   **`Equipo`**:
        -   `nombre`: `CharField` con `unique=True`.
        -   `logo`: `ImageField` (opcional, similar a la foto de la sede).
    -   **`Temporada`**:
        -   `nombre`: `CharField`.
        -   `liga`: `ForeignKey` a `Liga`.
        -   `modalidad`: `CharField` con opciones (`choices`) predefinidas: 'ida' (Una vuelta), 'ida_vuelta' (Doble vuelta).
        -   `estado`: `CharField` con opciones (`choices`): 'planeada', 'activa', 'finalizada', 'cancelada'.
    -   **`Jugador`**:
        -   `nombre`: `CharField`.
        -   `apellido_paterno`: `CharField`.
        -   `apellido_materno`: `CharField` (opcional).
        -   `fecha_nacimiento`: `DateField`.
        -   `equipo`: `ForeignKey` a `Equipo`.

### 2. Configuración del Entorno (`docker-compose.yml` y `settings.py`)

-   **Docker para PostgreSQL:**
    -   Se creará un archivo `docker-compose.yml` en la raíz del proyecto.
    -   Definirá un servicio `db` que utilizará la imagen oficial de `postgres`.
    -   Se configurará un volumen (`postgres_data`) para garantizar la persistencia de los datos entre reinicios del contenedor.
    -   Las credenciales y el nombre de la base de datos se gestionarán a través de variables de entorno en el `docker-compose.yml` y se leerán en `settings.py` (usando `os.getenv`).

-   **Configuración de Django (`settings.py`):**
    -   Se instalará `psycopg2-binary` para permitir la conexión con PostgreSQL.
    -   La configuración de `DATABASES` apuntará al servicio `db` de Docker (host: `db`, puerto: `5432`).
    -   Se configurarán `MEDIA_URL = '/media/'` y `MEDIA_ROOT = BASE_DIR / 'mediafiles'`. Esto le indicará a Django dónde guardar los archivos subidos y cómo servirlos durante el desarrollo.
    -   Se modificará el `urls.py` principal para que sirva los archivos de medios en modo `DEBUG`.

### 3. Administración (`core/admin.py`)

-   Para acelerar el desarrollo y cumplir con los criterios de aceptación, se registrarán todos los modelos (`Federacion`, `Liga`, `Temporada`, `Equipo`, `Jugador`, `Sede`) en el panel de administración de Django.
-   Se utilizará `admin.site.register(Modelo)` para cada uno. Esto proporcionará una interfaz CRUD funcional de inmediato para la gestión de los catálogos base.

### 4. Población de Datos (`core/management/commands/seed_data.py`)

-   Se creará un `management command` personalizado para poblar la base de datos.
-   El comando `seed_data` realizará las siguientes acciones en orden:
    1.  Creará una o más Federaciones.
    2.  Creará Ligas asociadas a esas federaciones.
    3.  Creará Equipos.
    4.  Creará Sedes.
    5.  Creará Temporadas para las ligas.
    6.  Creará Jugadores y los asociará a los equipos.
-   Este script será fundamental para tener un entorno de desarrollo consistente y para las pruebas de futuros incrementos. Se utilizará el método `get_or_create` de Django para evitar duplicar datos si el comando se ejecuta varias veces.
