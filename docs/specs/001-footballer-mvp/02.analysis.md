# Spec: 001 - Footballer MVP

## 2. Analysis

### 2.1. Análisis Técnico de Requerimientos

El objetivo es construir una aplicación web monolítica con Django, que sirva tanto la lógica de negocio como la interfaz de usuario. La arquitectura se centrará en el patrón Modelo-Vista-Template (MVT) de Django, aprovechando su ORM para la interacción con la base de datos PostgreSQL y su sistema de plantillas para renderizar el frontend con Bootstrap.

### 2.2. Stack Tecnológico

-   **Backend:** Python 3.x, Django 5.x.
-   **Base de Datos:** PostgreSQL.
-   **Frontend:** HTML5, CSS3, JavaScript, Bootstrap 5.x.
-   **Servidor de Desarrollo:** Servidor de desarrollo de Django.
-   **Control de Versiones:** Git.

### 2.3. Modelo de Datos (Entidades Principales)

El análisis del modelo de datos se basa en el archivo `migraciones.py` y el diagrama ER propuesto. Las entidades centrales son:

-   `Federacion`: Agrupa ligas.
-   `Liga`: Agrupa temporadas y equipos.
-   `Temporada`: Define un ciclo de competencia (ej. Apertura 2026) con sus propias reglas y equipos participantes.
-   `Equipo`: Catálogo central de equipos. Puede participar en múltiples ligas y temporadas.
-   `Jugador`: Catálogo de jugadores, asociados a un equipo.
-   `Arbitro`: Catálogo de árbitros con un flag para permisos de captura general.
-   `Sede`: Ubicaciones donde se juegan los partidos.
-   `Partido`: El corazón del sistema. Registra el enfrentamiento entre dos equipos, fecha, sede, árbitro, marcador y estado.
-   `EventoPartido`: Registra los eventos clave de un partido (goles, tarjetas).
-   `Usuario` y `Rol` (Grupos de Django): Para gestionar la seguridad y los permisos.

Tablas intermedias para relaciones N:M:

-   `LigaEquipo`: Asocia equipos a una liga.
-   `TemporadaEquipo`: Define los equipos que participan en una temporada específica.

### 2.4. Desglose de Features y Arquitectura

#### F01-F07: Gestión de Entidades Base (CRUDs)

-   **Análisis:** Estas features son operaciones CRUD (Crear, Leer, Actualizar, Eliminar) estándar.
-   **Solución Propuesta:** Se implementarán utilizando Vistas Basadas en Clases (CBV) de Django (`CreateView`, `UpdateView`, `ListView`, `DetailView`). Se puede aprovechar el `Django Admin` para una administración rápida inicial, pero las vistas personalizadas para los roles de administrador de liga serán necesarias.

#### F08, F15: Generación de Calendario y Cargas Masivas

-   **Análisis:** La generación de calendario es un proceso algorítmico (round-robin). Las cargas masivas desde Excel requieren validación robusta.
-   **Solución Propuesta:**
    -   **Generación de Calendario:** Un servicio o función en `services.py` que tome la lista de equipos y la configuración de la temporada para generar los objetos `Partido` y `Jornada`.
    -   **Cargas Masivas:** Utilizar la librería `pandas` o `openpyxl` para leer los archivos Excel. Se creará un "form" de Django que maneje la subida del archivo y un servicio de validación que procese el `DataFrame` de pandas, revise las reglas de negocio (duplicados, referencias inexistentes) y devuelva una previsualización y una lista de errores antes de confirmar la transacción en la base de datos.

#### F10-F11: Registro de Resultados y Eventos

-   **Análisis:** Esta es una operación crítica y transaccional. El marcador final debe ser consistente con los eventos de gol registrados.
-   **Solución Propuesta:** Se creará una vista específica para la captura de resultados. La lógica se encapsulará en un `formset` de Django para registrar los eventos (goles, tarjetas) dinámicamente. El guardado se ejecutará dentro de una transacción de base de datos (`transaction.atomic`) para garantizar que el `Partido` se actualice junto con todos sus `EventoPartido` de forma atómica.

#### F13-F14: Tablas de Clasificación y Goleadores

-   **Análisis:** Estas son consultas de agregación complejas que pueden ser costosas si no se optimizan.
-   **Solución Propuesta:**
    -   **Tabla General:** Se creará un método en el `Manager` del modelo `Temporada` o un servicio dedicado que utilice el ORM de Django con `annotate()` y `aggregate()` para calcular todas las estadísticas (puntos, goles, etc.) a partir de los partidos jugados. La consulta se ordenará directamente en la base de datos.
    -   **Tabla de Goleadores:** Similarmente, una consulta sobre `EventoPartido` agrupando por `jugador` y contando los goles. Se usará `Window functions` si es necesario para calcular el ranking compartido directamente en la base de datos para mayor eficiencia.

### 2.5. Roles y Permisos

-   **Análisis:** El sistema define roles claros con diferentes niveles de acceso.
-   **Solución Propuesta:** Se utilizará el sistema de autenticación y permisos de Django.
    -   Se crearán `Grupos` de Django que correspondan a los roles: `Administrador General`, `Administrador de Liga`, `Arbitro`.
    -   El permiso especial de "árbitro general" se manejará con el campo booleano `es_arbitro_general` en el modelo `Arbitro`.
    -   Las vistas se protegerán con `mixins` como `LoginRequiredMixin` y `PermissionRequiredMixin` o decoradores. La lógica de negocio validará el alcance (ej. un admin de liga solo puede ver sus temporadas).

### 2.6. Supuestos y Riesgos Técnicos

-   **Supuesto:** El volumen de datos por temporada es manejable para cálculos de tablas en tiempo real. Si el rendimiento se degrada, se podría explorar la desnormalización o el cacheo en una fase posterior.
-   **Riesgo:** La lógica de validación de los archivos Excel puede ser compleja. Se requiere una cobertura de pruebas unitarias exhaustiva para esta feature.
-   **Riesgo:** La generación del calendario (algoritmo round-robin) debe manejar correctamente un número par o impar de equipos.
