# Spec: 004 - Incremento 03: Gestión de Calendarios y Partidos

## 1. Requerimientos

### 1.1. Visión General

Este incremento se centra en la creación y gestión de los elementos necesarios para organizar los partidos de una temporada: los **árbitros**, las **sedes** (estadios), el **calendario** de juego, las **jornadas** y los **partidos** individuales. El objetivo es sentar las bases para poder registrar los resultados y eventos de cada encuentro.

### 1.2. Épica de Referencia

-   **Épica:** `Epica-Features-Footballer-03.md`
-   **Features Relevantes:**
    -   Gestión básica de árbitros.
    -   Gestión básica de sedes o estadios.
    -   Generación automática de calendario por temporada.
    -   Administración de jornadas y partidos.
    -   Sede obligatoria para todos los partidos.
    -   Asignación de árbitro antes de registrar resultado.
    -   Estados de partido.

### 1.3. Requerimientos Funcionales

#### RF-03.1: Gestión de Árbitros

-   **Descripción:** El sistema debe permitir a un `Administrador de Liga` gestionar un catálogo de árbitros.
-   **Criterios de Aceptación:**
    -   Se debe poder crear, editar y consultar árbitros.
    -   Un árbitro debe tener, como mínimo, los siguientes campos: `nombre`, `apellido_paterno`, `apellido_materno`, `fecha_nacimiento` y un campo booleano `activo`.
    -   La gestión se realizará a través del panel de administración de Django.

#### RF-03.2: Gestión de Sedes (Estadios)

-   **Descripción:** El sistema debe permitir a un `Administrador de Liga` gestionar un catálogo de sedes o estadios donde se jugarán los partidos.
-   **Criterios de Aceptación:**
    -   Se debe poder crear, editar y consultar sedes.
    -   Una sede debe tener, como mínimo, los siguientes campos: `nombre`, `ciudad`, `estado` y un campo booleano `activo`.
    -   La gestión se realizará a través del panel de administración de Django.

#### RF-03.3: Modelos para Jornadas y Partidos

-   **Descripción:** El sistema debe contar con los modelos de datos necesarios para estructurar un calendario de juegos en jornadas y partidos.
-   **Criterios de Aceptación:**
    -   Debe existir un modelo `Jornada` que se asocie a una `Temporada` y tenga un `numero` de jornada.
    -   Debe existir un modelo `Partido` que se asocie a una `Jornada`.
    -   Un `Partido` debe tener los siguientes campos:
        -   Relación con el equipo local (`EquipoTemporada`).
        -   Relación con el equipo visitante (`EquipoTemporada`).
        -   Relación con la `Sede` donde se jugará.
        -   Relación con el `Arbitro` asignado (puede ser nulo inicialmente).
        -   `fecha_hora` del partido.
        -   Un campo `estado` con valores predefinidos (ej: "Pendiente", "Programado", "Finalizado").

### 1.4. Requerimientos No Funcionales

-   **RNF-03.1:** Todos los nuevos modelos deben estar integrados en el panel de administración de Django.
-   **RNF-03.2:** El comando `seed_data` debe ser actualizado para crear datos de prueba para árbitros, sedes, jornadas y partidos para la temporada "Apertura 2024" de la "Liga MX".

### 1.5. Fuera del Alcance para este Incremento

-   Generación automática del calendario.
-   Registro de resultados (marcadores, goles, tarjetas).
-   Carga masiva de calendarios desde un archivo.
-   Validaciones complejas sobre los estados del partido.
