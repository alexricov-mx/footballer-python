# Spec: 004 - Incremento 03: Gestión de Calendarios y Partidos

## 2. Análisis Técnico

### 2.1. Resumen de la Solución

Para cumplir con los requerimientos, se crearán cuatro nuevos modelos en la app `core`: `Arbitro`, `Sede`, `Jornada` y `Partido`. Estos modelos se integrarán con los ya existentes (`Temporada`, `EquipoTemporada`) para formar la estructura del calendario de una liga.

La lógica de negocio será sencilla en este incremento, ya que nos centraremos en la creación de la estructura de datos y su administración a través del panel de Django.

### 2.2. Cambios en la Base de Datos

Se añadirán los siguientes modelos al archivo `core/models.py`:

1.  **`Arbitro`**:
    -   `nombre`: `CharField`
    -   `apellido_paterno`: `CharField`
    -   `apellido_materno`: `CharField`
    -   `fecha_nacimiento`: `DateField`
    -   `activo`: `BooleanField` (default: `True`)
    -   Será un modelo simple, sin relaciones foráneas por ahora.

2.  **`Sede`**:
    -   `nombre`: `CharField`
    -   `ciudad`: `CharField`
    -   `estado`: `CharField`
    -   `activo`: `BooleanField` (default: `True`)
    -   También será un modelo independiente en esta fase.

3.  **`Jornada`**:
    -   `numero`: `PositiveIntegerField`
    -   `temporada`: `ForeignKey` a `Temporada`.
    -   Se añadirá una constraint `UniqueConstraint` para asegurar que el `numero` de jornada sea único por `temporada`.

4.  **`Partido`**:
    -   `jornada`: `ForeignKey` a `Jornada`.
    -   `equipo_local`: `ForeignKey` a `EquipoTemporada`.
    -   `equipo_visitante`: `ForeignKey` a `EquipoTemporada`.
    -   `sede`: `ForeignKey` a `Sede`.
    -   `arbitro`: `ForeignKey` a `Arbitro` (permitiendo valores nulos, `null=True`, `blank=True`).
    -   `fecha_hora`: `DateTimeField`.
    -   `estado`: `CharField` con una lista de opciones (`choices`) para los estados definidos (ej. "Pendiente", "Programado", "Finalizado").
    -   Se añadirán `related_name` a las claves foráneas de `equipo_local` y `equipo_visitante` para evitar conflictos en la relación inversa desde `EquipoTemporada`.

### 2.3. Impacto en el Panel de Administración

-   Se registrarán los cuatro nuevos modelos en `core/admin.py`.
-   Para el modelo `Partido`, se configurará el `list_display` para mostrar información relevante como los equipos, la fecha, la jornada y el estado.
-   Se usarán `list_filter` y `search_fields` para facilitar la búsqueda y filtrado de partidos.
-   Se podría considerar un `Inline` de `Partido` dentro de la vista de `JornadaAdmin` para una gestión más fluida.

### 2.4. Actualización de los Datos de Prueba (`seed_data`)

-   El comando `seed_data` será extendido.
-   Se crearán varios `Arbitro` y `Sede` de ejemplo.
-   Se generará un calendario simple para la "Liga MX" - "Apertura 2024":
    -   Se crearán 17 jornadas.
    -   Para cada jornada, se crearán 9 partidos, asignando aleatoriamente los equipos locales y visitantes de la `EquipoTemporada` correspondiente.
    -   Se asignará una sede y un árbitro aleatorio a cada partido.
    -   Las fechas de los partidos se distribuirán a lo largo de un periodo de tiempo ficticio.

### 2.5. Riesgos y Consideraciones

-   **Conflicto de `ForeignKey` en `Partido`**: Es crucial definir `related_name` distintos para `equipo_local` y `equipo_visitante` para que Django pueda diferenciar las relaciones inversas. Por ejemplo: `related_name='partidos_local'` y `related_name='partidos_visitante'`.
-   **Lógica del Seeder**: La generación de partidos en el seeder debe asegurar que un equipo no juegue contra sí mismo y, idealmente, que no juegue más de una vez por jornada. Dado que es solo para datos de prueba, una asignación aleatoria simple será suficiente por ahora, pero se debe tener cuidado de no crear datos inconsistentes.
-   **Migraciones**: Los cambios son aditivos (nuevos modelos), por lo que el proceso de migración debería ser directo y sin complicaciones.
