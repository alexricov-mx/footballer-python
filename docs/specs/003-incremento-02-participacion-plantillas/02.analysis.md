# Spec: 003 - Incremento 02: Participación y Plantillas

## 2. Análisis y Diseño

Este documento presenta el análisis técnico y las decisiones de diseño para implementar los requisitos de gestión de participaciones y plantillas.

### Análisis de Requisitos

1.  **Necesidad de una tabla intermedia para Equipo-Temporada:** El requisito de que un equipo pueda participar en múltiples temporadas (`RF-02.1`) implica una relación de muchos a muchos entre `Equipo` y `Temporada`. Crearemos un modelo explícito para esta relación, `EquipoTemporada`, que servirá como tabla intermedia. Esto nos permitirá añadir metadatos a la participación en el futuro si fuera necesario (por ejemplo, puntos de penalización, estado de inscripción, etc.).

2.  **Necesidad de una tabla para la Plantilla:** El requisito de que un jugador tenga un dorsal y una posición específicos para una temporada (`RF-02.2`) indica que la relación no es directamente entre `Jugador` y `Equipo`, sino entre `Jugador` y la participación del `Equipo` en una `Temporada`. Por lo tanto, crearemos el modelo `PlantillaJugador` que se vinculará con `EquipoTemporada`.

3.  **Impacto en Modelos Existentes (`RF-02.3`):**
    -   **Modelo `Jugador`:** La clave foránea `equipo` debe ser eliminada. Esto desacopla al jugador de un único equipo y lo convierte en una entidad más global, que puede ser "fichado" por diferentes equipos en diferentes temporadas.
    -   **Modelo `Temporada`:** La clave foránea `liga` debe ser eliminada. La liga a la que pertenece una temporada se puede inferir a través de los equipos que participan en ella. Sin embargo, para mantener la simplicidad y una referencia directa, **se mantendrá la relación `Temporada` -> `Liga` por ahora**. Esto evita consultas complejas y asume que una temporada pertenece a una única liga, lo cual es coherente con el dominio del problema. La refactorización se pospondrá si el modelo de negocio evoluciona.

### Diseño de la Base de Datos

Se proponen los siguientes modelos nuevos y modificaciones:

#### Nuevos Modelos

1.  **`EquipoTemporada` (Participación):**
    -   `temporada`: `ForeignKey` a `Temporada`.
    -   `equipo`: `ForeignKey` a `Equipo`.
    -   `creado`, `actualizado`: Timestamps de auditoría.
    -   **Constraint:** `unique_together = ('temporada', 'equipo')` para evitar inscripciones duplicadas.

2.  **`PlantillaJugador` (Plantilla):**
    -   `equipo_temporada`: `ForeignKey` a `EquipoTemporada`.
    -   `jugador`: `ForeignKey` a `Jugador`.
    -   `dorsal`: `PositiveIntegerField`.
    -   `posicion`: `CharField` con `choices` (Portero, Defensa, Mediocampista, Delantero).
    -   `creado`, `actualizado`: Timestamps de auditoría.
    -   **Constraint:** `unique_together = ('equipo_temporada', 'dorsal')` para asegurar dorsales únicos por plantilla.
    -   **Constraint:** `unique_together = ('equipo_temporada', 'jugador')` para evitar que un jugador esté dos veces en la misma plantilla.

#### Modificaciones a Modelos Existentes

1.  **`Jugador`:**
    -   Se elimina el campo `equipo = ForeignKey(Equipo, ...)`.
    -   Se mantiene `nombre`, `apellido`, `activo`, etc., como datos biográficos del jugador.

2.  **`Temporada`:**
    -   Se mantiene el campo `liga = ForeignKey(Liga, ...)`. La decisión es mantener esta relación explícita para simplificar las consultas y la lógica de negocio en esta etapa.

### Diseño del Panel de Administración (`RF-02.4`)

Para una gestión fluida, se implementarán `inlines` en el panel de administración:

1.  **En la vista de `Temporada`:**
    -   Se usará un `TabularInline` para mostrar y gestionar los `EquipoTemporada` (los equipos participantes). Esto permitirá añadir o quitar equipos directamente desde la página de una temporada.

2.  **En la vista de `EquipoTemporada`:**
    -   Se usará un `TabularInline` para mostrar y gestionar los `PlantillaJugador` (la plantilla de jugadores). Esto permitirá conformar el roster (añadir jugadores, asignarles dorsal y posición) directamente desde la página de participación del equipo.

3.  **En la vista de `Equipo`:**
    -   Se podría añadir un `inline` para ver en qué temporadas ha participado el equipo, pero se considera secundario para esta fase.

### Impacto en el Comando `seed_data`

El script `seed_data.py` deberá ser reescrito casi en su totalidad para reflejar la nueva estructura:
1.  **Orden de Creación:**
    1.  Crear `Federacion`, `Liga`, `Sede`.
    2.  Crear `Equipo` y `Jugador` como entidades independientes.
    3.  Crear `Temporada`.
    4.  Crear las instancias de `EquipoTemporada` para inscribir equipos en temporadas.
    5.  Crear las instancias de `PlantillaJugador` para asignar jugadores a las plantillas de los equipos en sus respectivas temporadas.
2.  **Lógica:** La lógica de asignación de jugadores a equipos será más compleja, ya que ahora pasa por las tablas intermedias.
