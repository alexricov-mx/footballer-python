# Spec: 003 - Incremento 02: Participación y Plantillas

## 1. Requerimientos

Este documento detalla los requisitos funcionales y no funcionales para la gestión de la participación de equipos en temporadas y la conformación de sus plantillas de jugadores.

### Contexto

Basado en la épica general (`Epica-Features-Footballer-06.md`), este incremento se enfoca en establecer las relaciones clave entre equipos, temporadas y jugadores. Actualmente, un `Jugador` pertenece a un `Equipo` de forma general, pero no existe una vinculación a una `Temporada` específica. De manera similar, un `Equipo` no está formalmente inscrito en una `Temporada`.

El objetivo es modelar que un `Equipo` participa en una `Temporada` con una `Plantilla` (roster) específica de jugadores para esa competición.

### Requisitos Funcionales

#### RF-02.1: Modelo de Participación de Equipos
-   Se debe crear un modelo que represente la participación de un `Equipo` en una `Temporada` (`EquipoTemporada`).
-   Esta relación debe registrar qué equipos están inscritos en qué temporadas.
-   Un equipo debe poder participar en múltiples temporadas, incluso de diferentes ligas, simultáneamente.
-   La interfaz de administración de Django debe permitir gestionar estas participaciones.

#### RF-02.2: Modelo de Plantilla de Jugadores
-   Se debe crear un modelo que represente la plantilla de un equipo para una temporada específica (`PlantillaJugador`).
-   Este modelo debe vincular a un `Jugador` con la participación de un `Equipo` en una `Temporada` (`EquipoTemporada`).
-   Debe registrar el `dorsal` (número de camiseta) que el jugador usará en esa temporada, el cual debe ser único dentro de la misma plantilla.
-   La posición del jugador (`Portero`, `Defensa`, etc.) se registrará en este modelo, permitiendo que un jugador pueda tener diferentes posiciones o roles en distintas temporadas o equipos.

#### RF-02.3: Refactorización de Modelos Existentes
-   El modelo `Jugador` debe ser modificado para eliminar la relación directa y obligatoria con `Equipo`. Un jugador existirá en el sistema de forma más genérica y se asociará a un equipo a través de la `PlantillaJugador`.
-   El modelo `Temporada` debe ser modificado para eliminar la relación directa con `Liga`, ya que esta relación ahora estará implícita a través de la participación de los equipos.

#### RF-02.4: Gestión en el Panel de Administración
-   El panel de administración de Django debe ser actualizado para permitir una gestión intuitiva de las nuevas relaciones:
    -   Al ver una `Temporada`, se debería poder ver y gestionar los `Equipos` que participan en ella.
    -   Al ver la participación de un `Equipo` en una `Temporada`, se debería poder gestionar la `Plantilla` de jugadores (añadir, eliminar, modificar dorsal y posición).

### Criterios de Aceptación

-   [ ] Se pueden crear, leer, actualizar y eliminar las participaciones de equipos en temporadas desde el panel de administración.
-   [ ] Se puede conformar una plantilla de jugadores para un equipo en una temporada específica, asignando dorsales y posiciones.
-   [ ] El sistema impide asignar el mismo dorsal a dos jugadores en la misma plantilla.
-   [ ] El modelo `Jugador` ya no depende directamente de un `Equipo`.
-   [ ] El panel de administración refleja la nueva estructura de datos de forma clara y funcional.
-   [ ] Los datos creados por el comando `seed_data` se adaptan a la nueva estructura de modelos.
