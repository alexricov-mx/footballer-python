# Spec: 003 - Incremento 02: Participación y Plantillas

## 3. Plan de Implementación

Este documento desglosa las tareas técnicas necesarias para llevar a cabo el diseño propuesto en `02.analysis.md`.

### Tareas de Desarrollo

#### Tarea 1: Modificación de Modelos (`core/models.py`)

-   **[ ] Tarea 1.1: Modificar el modelo `Jugador`**
    -   **Acción:** Eliminar el campo `equipo = models.ForeignKey(Equipo, ...)`.
    -   **Impacto:** Desacopla al jugador de un equipo fijo.

-   **[ ] Tarea 1.2: Crear el modelo `EquipoTemporada`**
    -   **Acción:** Añadir la nueva clase `EquipoTemporada` con los campos `temporada` (FK a `Temporada`), `equipo` (FK a `Equipo`) y la restricción `unique_together`.

-   **[ ] Tarea 1.3: Crear el modelo `PlantillaJugador`**
    -   **Acción:** Añadir la nueva clase `PlantillaJugador` con los campos `equipo_temporada` (FK a `EquipoTemporada`), `jugador` (FK a `Jugador`), `dorsal` y `posicion`.
    -   **Acción:** Añadir las restricciones `unique_together` para `(equipo_temporada, dorsal)` y `(equipo_temporada, jugador)`.

#### Tarea 2: Migración de la Base de Datos

-   **[ ] Tarea 2.1: Generar las nuevas migraciones**
    -   **Acción:** Ejecutar el comando `python manage.py makemigrations core`.
    -   **Resultado esperado:** Django detectará los cambios (un campo eliminado y dos modelos nuevos) y generará un nuevo archivo de migración.

-   **[ ] Tarea 2.2: Aplicar las migraciones**
    -   **Acción:** Ejecutar el comando `python manage.py migrate`.
    -   **Observación:** Antes de esto, es recomendable limpiar la base de datos para evitar conflictos con los datos existentes. Se puede hacer bajando el contenedor con `podman-compose down -v`.

#### Tarea 3: Actualización del Panel de Administración (`core/admin.py`)

-   **[ ] Tarea 3.1: Crear `inlines` para la gestión de plantillas y participaciones**
    -   **Acción:** Crear una clase `PlantillaJugadorInline(admin.TabularInline)`.
    -   **Acción:** Crear una clase `EquipoTemporadaInline(admin.TabularInline)` que a su vez utilice el `PlantillaJugadorInline`.

-   **[ ] Tarea 3.2: Registrar los nuevos modelos y actualizar los existentes**
    -   **Acción:** Registrar los modelos `EquipoTemporada` y `PlantillaJugador`.
    -   **Acción:** Modificar el registro de `TemporadaAdmin` para que incluya el `EquipoTemporadaInline`.
    -   **Acción:** Crear un `EquipoTemporadaAdmin` que incluya el `PlantillaJugadorInline` para gestionar las plantillas desde la participación.

#### Tarea 4: Actualización del Comando de Seeding (`core/management/commands/seed_data.py`)

-   **[ ] Tarea 4.1: Refactorizar el script `seed_data.py`**
    -   **Acción:** Modificar la lógica de creación de datos para seguir la nueva estructura:
        1.  Crear entidades base (`Federacion`, `Liga`, `Sede`, `Equipo`, `Jugador`).
        2.  Crear `Temporada`.
        3.  Crear `EquipoTemporada` para inscribir equipos.
        4.  Crear `PlantillaJugador` para asignar jugadores a las plantillas.
    -   **Observación:** Esto requerirá una reescritura significativa del método `handle`.

-   **[ ] Tarea 4.2: Probar el comando de seeding**
    -   **Acción:** Ejecutar `python manage.py seed_data` para verificar que la base de datos se puebla correctamente sin errores.

#### Tarea 5: Verificación Final

-   **[ ] Tarea 5.1: Crear un nuevo superusuario**
    -   **Acción:** Ejecutar `python manage.py createsuperuser` (después de limpiar la BD y migrar).

-   **[ ] Tarea 5.2: Verificación manual en el Panel de Administración**
    -   **Acción:** Iniciar el servidor (`python manage.py runserver`).
    -   **Acción:** Acceder al panel de admin y verificar:
        -   Que se pueden añadir equipos a una temporada.
        -   Que se puede construir la plantilla para un equipo en una temporada, asignando dorsales y posiciones.
        -   Que las validaciones de unicidad (dorsal y jugador por plantilla) funcionan correctamente.
        -   Que la navegación a través de los `inlines` es intuitiva.
